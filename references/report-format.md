# 金字塔研报格式

读者只有 2 分钟。信息架构的目标是：**10 秒扫完重点，想深入再往下滚**。所以正文极简、结论先行，六段全文沉底到附录。

## 为什么这样排

一份研报最容易犯的错是「把深度做成全文平铺」——5 条必读 × 每条六段 ≈ 3000 字堆在正文，金字塔顶就被压塌了。读者要的不是事件复述，而是**「这事对我意味着什么 + 我这周该做什么」**。所以：

- 正文每条必读压成 **3 行**：标题 + 一句话结论（so-what）+ 首要行动。
- 完整的六段分析（背景/影响/竞品/提效/机会/行动）移到**文末附录**，按需展开。
- 低分条目（≤5）不进正文，只在尾注给个总数 + 去哪看。

## 模板（严格遵循）

```markdown
# AI 情报{日报|周报|月报} · {period}

_☕ 正文约 N 分钟读完 · 深度解读见文末附录_

## ⚡ 今日速览

{恰好 3 句话，每句一个重点，合计 ≤120 字。直接给结论，不要「今天有以下几条」这种引语。}

## 🔥 今日必读

**1. [{中文标题≤30字}]({url})** `{importance}/10` · {来源}
⚡ {so-what：这件事对 {lens} 最重要的含义，≤40 字，直击要害}
👉 {首要行动：本周可做的最具体一件事}

**2. [...]({url})** `9/10` · {来源}
⚡ ...
👉 ...

（≥8 分的 Top 3-5 条；若不足，放宽到 ≥7 分取前 3）

## 📌 值得关注

- **[{标题}]({url})** `7` · {内容分类} · {来源}
- **[{标题}]({url})** `6` · {内容分类} · {来源}

（6-7 分，每条仅 1 行，最多 8 条；摘要不进正文，读者要细节自己点链接）

## 🔧 GitHub 高星开源盘点（可选板块）

> 触发：**PM / 工程师 / 研究 lens 默认带上**（PM 看产品机会与需求趋势、工程看技术选型与集成、研究看方法新意，详见 `lenses.md`）；投资 lens 当日开源有料时可选。数据来自 `github_stars` 源（GitHub Search API 按 stars 降序，多 topic 合并去重）。

**① [{owner/repo}]({url})** ⭐{stars} · {语言} · `{一句话定位标签}`
- 🎯 **产品**：{解决谁的什么问题、能拿来做什么，1-2 句}
- 🔧 **技术**：{技术栈 / 架构亮点，1 句}
- 💡 **为何现在看**：{与当期主线 / 趋势的呼应，1 句}

（每个项目 3 行解读；末尾加一句「给 PM / 工程的归纳」点出共同趋势。这是 ai-radar 区别于纯新闻聚合的「产研提效」抓手）

---

_本期共收录 **{total}** 条 · 其余 {rest} 条见各来源 · 降级源 {n} 个_

## 📚 附录 · 深度解读

> 来源：[{来源}]({url}) · 重要度 {importance}/10

- **事件背景**：{2-3 句}

- **竞品对位**：{2-3 句}
- **{lens}视角启示**：{结合视角的可落地动作}

  - {动作1}

...
```

## 周报 / 月报的差异

周报、月报不是「日报变长」，而是**趋势提炼**。把当期全部条目按 8 类归并后：

- 开头换成「本期大事记」（3-5 句概括最重要的行业走向，而非逐条罗列）。
- 加一节「📈 趋势研判」：模型 / 工具 / 开源 / 产品商业四个维度，各 2-4 句讲清这一周/月的方向。
- 加一节「🔭 下期关注」：3 条值得盯的方向。
- 必读区改为「本期重大事件」（≥8 分全列），附录同样沉底。

## 写作要点

- **so-what 是灵魂**：每条必读的 ⚡ 行决定读者要不要展开。写「这事改变了什么、你能利用什么」，不要写「X 公司发布了 Y」。
- **行动要具体到能本周执行**：「评估 X」太虚；「本周用 X 在现有产品的 Y 场景跑个 benchmark，对比成本」才有用。
- **诚实优先**：信息少就说少，不拔高重要度凑必读；抓取有失败就在尾注标注。
- **数字用阿拉伯数字**，公司/产品用规范名，便于读者检索。
